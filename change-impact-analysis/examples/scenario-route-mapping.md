# 路由映射表生成

独立于变更分析，手工梳理或更新项目的路由 → 菜单路径映射关系。

## 核心原则：菜单路径 ≠ 路由模块

**菜单路径**的定义是：**用户在 App 里点击哪个菜单才能触达这个页面**，而不是「这个页面的文件在哪个目录」。

### 两类路由的本质区别

| 类型 | 定义 | 菜单路径取法 | 示例 |
|------|------|------------|------|
| **直达菜单路由** | 用户点击导航/菜单直接进入 | 该菜单项本身 | `/users` → 用户管理 |
| **子流程路由** | 由另一个业务页面「推入」的流程页 | **触发入口**所在的菜单 | `/order/refund` → 订单管理（若从订单列表页发起退款） |
| **多入口路由** | 可从多个不同菜单触达 | 列出所有入口 | `/payment` → 多入口（订单管理 / 购物车 / 会员中心） |

### 典型错误：按文件目录判断

```
❌ 错误推断：
  文件: src/pages/order/refund/RefundPage.vue
  → 所属模块 order → 菜单路径：订单管理  ← 可能错误！

✅ 正确推断：
  搜索谁在调用 /order/refund：
  → 发现从「商品管理」页的下架操作中会跳入
  → 菜单路径：商品管理（而非订单管理）
```

## 菜单路径推断策略

按以下优先级确定菜单路径：

1. **路由菜单标题字段含中文** — 直接作为页面标题（React 项目为 `crumbs`，Vue 项目为 `meta.title`，详见对应指南）
2. **`src/app.json` 中的 `title`** — 自动读取页面注册表中对应 `path` 的 `title`
3. **组件文件标题回退** — 路由无中文标题时，读取 `component` 指向的文件提取标题
4. **路径分段兜底** — 供人工审核补充，**报告中必须替换为中文**

### 如何判断一个路由是子流程路由

在代码中搜索该路由 path，如果它被**多个不同模块**的文件 push 或 replace，说明它是子流程路由或多入口路由，需要人工确认菜单归属后在报告中标注。

```bash
# 以 /order/refund 为例：搜索谁在导航到它
grep -r "order/refund" src/ --include="*.vue" --include="*.ts" --include="*.tsx" -n \
  | grep -v "router\.\(ts\|js\)"
```

## 梳理步骤

1. **读取路由配置文件**（`router.ts` / `routes.ts` 等），提取所有 `path` 与 `meta.title`（或 `crumbs`）。
2. **对无中文标题的路由**，读取 `component` 引用的组件文件，尝试提取中文标题（`defineOptions.name` 含中文、导航栏 `title` 属性等）。
3. **搜索子流程路由入口**：对无法从路由自身判断归属的路由，用 `grep` 搜索其 path 被哪些页面 push。
4. **确认多入口路由**：如果同一 path 被多个不同菜单触达，菜单路径格式为 `多入口（A / B / C）`。

## 注意事项

- 约定式路由（Next.js `app/`、Nuxt `pages/`）需手动梳理文件结构与菜单对应关系
- 动态路由参数（如 `:id`）在匹配时去除后缀，仅匹配前缀段
- 嵌套路由（`children`）展平后子路由 path 会拼接父路由 path
- 路由 title 无中文时尝试从组件文件自动提取（见 SKILL.md）
